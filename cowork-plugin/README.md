# Talent-Augmenting OS — Claude Cowork plugin

The TAOS coaching layer, packaged for [Claude Cowork](https://www.anthropic.com/product/claude-cowork).

This is a sibling of the Claude Code plugin in [`../plugin/`](../plugin/). Same MCP
server, same coaching behaviour, different delivery mechanism, because Cowork and
Claude Code expose different surfaces to a plugin.

## What's in the bundle

```
cowork-plugin/
├── .claude-plugin/plugin.json   plugin manifest
├── .mcp.json                    registers the remote TAOS MCP server
├── protocols/                   hand-edited source: per-skill frontmatter + protocol body
│   ├── talent-assess.md
│   ├── talent-coach.md
│   └── talent-update.md
├── skills/                      GENERATED by _build_skills.py — do not hand-edit
│   ├── talent-assess/SKILL.md
│   ├── talent-coach/SKILL.md
│   └── talent-update/SKILL.md
├── _build_skills.py             assembles skills/ from protocols/ + the system prompt
├── INSTALL.md                   end-user install steps
└── README.md                    this file
```

## Design: why the skills are self-contained

The Claude Code plugin gets ambient coaching for free: a `SessionStart` hook
(`../plugin/hooks/inject-tal-layer.py`) injects the TAOS system prompt and the
user's profile into every session before turn one.

Cowork does **not** run plugin-scoped hooks. It spawns Claude with user-scope
settings only, so a bundled `hooks/` directory is silently ignored
([anthropics/claude-code#27398](https://github.com/anthropics/claude-code/issues/27398)).
That leaves three ways to get the system prompt in front of the model, and only
one of them needs zero manual setup:

| Mechanism | Manual step? | Reliable in Cowork? |
|---|---|---|
| `SessionStart` hook | none | **No** — hooks don't fire |
| Paste into Cowork Global Instructions | user pastes ~19k chars | yes, but rejected: too much friction |
| **Skill body carries the system prompt** | **none** | **yes** |

So every skill in this bundle is self-contained. `_build_skills.py` concatenates
the full TAOS system prompt (`../plugin/tal-system-prompt.md`) ahead of each
skill's protocol. Invoking any one skill primes the model with the complete
coaching layer for the rest of that conversation. The user just says "run my
TAOS assessment" (or anything that matches a skill `description`), and from that
turn on Claude is operating as their coach.

Trade-off: each skill invocation reloads ~19k characters of system prompt. If a
user invokes two skills in one conversation, that context is loaded twice. Tokens
are cheap and the model handles the redundancy fine, but it is not free. The
alternative — a shared init skill the others defer to — was rejected because the
model can't reliably introspect whether the system prompt is already in context.

## Differences from the Claude Code plugin

| | Claude Code plugin (`../plugin/`) | Cowork plugin (this) |
|---|---|---|
| Ambient system prompt | `SessionStart` hook | baked into each skill body |
| Profile storage | local filesystem (`profiles/`), hook resolves it | hosted DB via MCP; optional local copy if a workspace folder is linked |
| MCP server | same remote endpoint | same remote endpoint |
| Skills | filesystem-first (`Glob` for `profiles/*.md`) | MCP-tool-first (sandboxed VM has no `profiles/` dir) |
| `allowed-tools` frontmatter | explicit, plugin-namespaced tool names | omitted — Cowork's tool namespacing is unverified; skills inherit the toolset |

## Maintenance

`skills/` is generated. After editing either source, regenerate:

```sh
python3 cowork-plugin/_build_skills.py
```

Sources:
- `protocols/<name>.md` — the per-skill protocol (this is what you hand-edit)
- `../plugin/tal-system-prompt.md` — the shared TAOS system prompt

Commit the regenerated `skills/` alongside the source change.

## Known unknowns (verify with a real Cowork account)

These could not be confirmed against Anthropic docs and need a hands-on pass:

1. **Exact install path.** Marketplace listing vs. zip upload vs. GitHub URL — see `INSTALL.md` for the current best guess.
2. **`.mcp.json` auto-registration.** Whether Cowork picks up the bundled MCP server automatically, or the user has to add the connector by URL separately.
3. **Tool namespacing.** What prefix Cowork gives MCP tools from a bundled server. The skills reference tools by bare logical name to sidestep this; if Cowork needs explicit `allowed-tools`, that frontmatter has to be added.
4. **`description`-based auto-invocation.** Whether Cowork reliably self-selects a skill from its `description` without the user naming it. The descriptions here are written to maximise that, but it is unverified.

When the active pilot reaches a user with Cowork access, run the four flows
(assess, coach, update, profile fetch) and fold the findings back into this bundle.
